1. ChatCut Agent 的公开证据

ChatCut 官方文档直接把它定义成 web-based AI video editing agent：用户上传素材或从零创建项目，用自然语言描述目标，Agent 负责粗剪、找高光、生成字幕、生成动效、生成音乐等。它还写到当前的素材理解主要基于转写文本，最适合 talking head 和 interview footage，视觉分析仍在后续计划里。

它的编辑器不是单纯聊天框，而是五块组合：Agent、Player、Media、Timeline、Menu。Agent 区负责自然语言请求，比如剪视频、生成 motion graphics、下载媒体、写脚本；Media 区会自动同步上传和 AI 生成素材；Timeline 区仍然保留手动拖拽、剪刀裁切；Menu 有 undo、redo、save version、export。

功能页暴露了很关键的执行痕迹：用户说“让这段更顺、更紧凑，加动效和音乐”，Agent 先 read_script，判断 repeated takes、fillers、silences，然后 apply_script 执行剪辑，再 apply_script 加 motion graphics，再生成音乐并放到人声下面。这个 trace 说明它不是一次性生成成片，而是 读状态 → 判断 → 调工具 → 写 timeline patch。

它反复强调“real multi-track timeline”：自然语言会被路由到正确工具，最后落到真实时间线，而不是模板黑盒。官方也说“transcript is the timeline”：每个词带时间戳，删一句话，对应视频和音频也被删掉。

它和 CapCut 的对比页说得更直白：CapCut 是 template-driven，ChatCut 是 conversational AI agent；ChatCut 的强项是 multi-step agent execution，比如“加 intro title、删 0:15 的尴尬停顿、全片加字幕、加背景音乐”这种多步骤任务一次说完即可执行。

ChatCut 的 motion graphics 文档进一步暴露了 Agent 的“网页理解 + 资源抽取 + 模板组装”能力：用户给一个品牌网站，Agent 会浏览网页、下载 logo 和图片、理解卖点和品牌色，然后选择 motion graphics template，组装 promo 视频。

2. 反向推演：ChatCut Agent 的核心架构

可以把 ChatCut 拆成 7 层。

1）Project State：项目状态层

Agent 首先需要一个统一项目状态，而不是直接操作 mp4。

这个状态大概包括：

{
  "project_id": "xxx",
  "assets": [],
  "transcripts": [],
  "timeline": {
    "tracks": ["MG", "V2", "V1", "A1", "MUS", "CAPTIONS"],
    "clips": []
  },
  "versions": [],
  "export_settings": {}
}

ChatCut 的页面里出现了 MG、V1、A1、MUS 等轨道，而且所有生成的视频、图片、音乐、字幕、动效都进入同一个 editor / media pool / timeline，不需要来回导入导出。

设计原则：Timeline 是唯一真相源。

Agent 不能只输出“建议你剪哪里”，必须输出可执行的 timeline 修改。

2）Media Index：素材索引层

ChatCut 最先做的是把素材变成可检索对象。

公开资料显示它会：

自动转写视频；
区分说话人；
给句子或词加时间戳；
支持 100+ 语言；
用 transcript 驱动时间线编辑。

字幕页还写到 ChatCut 做 word-level timestamps、speaker identification、per-word highlighting，删 transcript 中的词可以精准删掉对应视频。

所以反推它的素材索引至少有三类：

{
  "semantic_index": "这段在讲什么",
  "time_index": "这句话/这个词在几秒到几秒",
  "speaker_index": "谁说的"
}

设计原则：先把视频变成可搜索、可重排、可引用的文本块。

这也是你用 OpenClacky 复刻时最重要的第一步。

3）Agent Planner：剪辑计划层

ChatCut 不是用户说一句话就直接渲染，而是先把请求拆成任务。

例子：

用户：把这个采访剪成 60 秒高光，加字幕、加音乐、做竖屏。

Agent 内部应该拆成：

1. 读取项目和逐字稿
2. 找高光片段
3. 生成剪辑结构
4. 生成 cut list
5. 应用到 timeline
6. 生成字幕
7. 加音乐并 duck 到人声下面
8. 重构为 9:16
9. 生成预览

官方博客也强调了 prompt-to-edit 的工作流：上传后自动转写、分离说话人、打时间戳；用户用自然语言给结构，比如按 topic 分组、找 soundbite、按 narrative 组织 timeline；然后再 review 和 fine-tune。

设计原则：Agent 先做剪辑方案，再执行剪辑动作。

这点很关键。OpenClacky 不能让模型直接“猜 ffmpeg 命令”，而是要先产出结构化 plan。

4）Tool Router：工具路由层

ChatCut 官方说用户不用选菜单，直接说需求，系统会把需求 route 到 right tool。

反推工具路由大概是：

剪停顿 → silence detector + timeline patch
删口癖 → transcript word timestamp + cut patch
加字幕 → ASR/caption engine + caption style engine
加动效 → motion graphics generator
加音乐 → music generator + audio mix
生成 B-roll → video/image generation
网页转广告 → browser + asset extractor + script writer + timeline assembler
导出 → renderer/exporter

ChatCut 的音频降噪页也暴露了一个多工具排序原则：如果用户说“清理音频、加字幕、加背景音乐”，Agent 会先降噪，因为字幕依赖清晰音频，然后再生成字幕，再加音乐。

设计原则：工具调用必须有依赖顺序，不是并列执行。

5）Timeline Patch：时间线修改层

最关键的是 apply_script。

ChatCut 页面中多次出现 apply_script：剪掉 dead air、压缩 pause、加 title card、加 lower-third、加 music bed，最后都落到 timeline。

所以它的内部很可能不是让 LLM 直接操作 DOM，而是生成某种 timeline patch：

{
  "op": "cut_segments",
  "track": "V1",
  "segments": [
    {"start": 0.0, "end": 0.8, "reason": "leading dead air"},
    {"start": 12.9, "end": 14.7, "reason": "trailing okay"}
  ]
}

或者：

{
  "op": "add_motion_graphic",
  "track": "MG",
  "template": "quote_lower_third",
  "start": 1.2,
  "duration": 4.5,
  "props": {
    "text": "核心观点",
    "style": "midnights"
  }
}

设计原则：Agent 输出 patch，不直接破坏原素材。

这保证可撤销、可版本管理、可复审。

6）Editable Generation：可编辑生成层

ChatCut 的 motion graphics 不是简单生成一段扁平视频。文档写得很明确：生成的是 editable animation component，用户可以改颜色、文字、字体、时长、布局、图片、视频等属性；不满意也可以继续让 Agent “把 logo 放大”“改成蓝色”。

这说明它的动效生成大概率不是纯视频生成，而是：

自然语言
→ 模板 / 代码化组件
→ props
→ 实时预览
→ timeline layer
→ 可导出透明 ProRes 4444

文档还写到 motion graphics 可以导出 ProRes 4444 alpha，方便放进 Premiere、Final Cut、DaVinci Resolve。

设计原则：生成物要保留结构化参数，不能只输出不可编辑 mp4。

这对 OpenClacky 很重要：用 Remotion / HTML / SVG / ASS 做可编辑组件，比直接生成视频更像 ChatCut。

7）Human-in-the-loop：人类协同层

ChatCut 不是完全黑盒。它保留 timeline、player、media、version、manual edit。官方文档写到用户可以手动拖拽素材进 timeline，用剪刀裁切，并且支持保存版本、回滚版本、导出。

Prompt-to-edit 博客也强调 review & fine-tune：用户可以拖动 transcript blocks 来测试新想法，timeline 立即更新；同时支持 transcript、字幕、XML 等导出。

LinkedIn 页面还提到，ChatCut 的 agent 会问用户想要什么、在编辑前检查，并为每个 clip 给 timeline，让用户继续 tweak、导出视频或 XML。

设计原则：Agent 做 first cut，人做 final decision。

这也是它和“一键生成视频”类工具的本质区别。

3. ChatCut Agent 的 12 条设计原则
原则 1：Agent 是主入口，不是边缘功能

ChatCut 的定位是“video editing agent”，不是“视频编辑器里加了 AI 功能”。官方明确说它是 agent-first，不是 turning captions on/off 这种单功能执行。

OpenClacky 复刻时应该把入口设计成：

用户意图 → Agent → Skill → timeline patch → preview

而不是：

用户点按钮 → 调一个 ffmpeg 命令
原则 2：真实时间线优先

ChatCut 所有功能都回到真实多轨时间线：视频生成、字幕、音乐、动效、配音、B-roll 都成为 timeline 上的 layer。

OpenClacky 也应该先定义：

timeline.json / project.json / edit_decisions.json

再做渲染。

原则 3：Transcript 是视频的语义操作系统

ChatCut 的关键不是“识别字幕”，而是把 transcript 当成剪辑界面。每个词有时间戳，删词就是删视频。

OpenClacky 应该优先复刻：

视频 → ASR → word timestamp → transcript block → cut list → timeline patch

这比先做花哨 UI 更重要。

原则 4：Agent 先读项目，再动手

页面里出现 read_script、read_project，说明 ChatCut 的 Agent 在执行前会读取项目和脚本状态，而不是凭空生成。

OpenClacky Skill 应该统一有：

read_project
read_transcript
read_timeline
read_assets

不要让每个 Skill 自己乱扫文件。

原则 5：复杂请求拆成多工具链

ChatCut 能把“一句话多动作”拆成降噪、字幕、音乐、动效、导出等多个工具动作。官方文档也展示了“生成 10 秒产品镜头 + lower third + 背景音乐 + 字幕”这种单 prompt 多 action 的模式。

OpenClacky 应该内置一个 planner：

intent_parse → dependency_graph → skill_sequence → execute → verify
原则 6：自动化的是 first cut，不是剥夺控制权

ChatCut 强调它不是 AI black box，因为它给完整 timeline editor，用户能控制 edit 怎么做。

OpenClacky 对外也应该讲：

不是一键生成不可控视频，而是生成可审计、可回滚、可继续改的剪辑工程。
原则 7：生成内容必须“落地到项目”，不能只下载文件

ChatCut 的生成视频、配音、音乐、SFX 都强调自动放到 timeline 或 media pool，不需要 download/import。

OpenClacky 复刻时，每个生成类 Skill 都应该输出：

{
  "asset_path": "...",
  "timeline_patch": "...",
  "preview_path": "..."
}

而不是只输出一个文件。

原则 8：可编辑组件优先于扁平视频

Motion graphics 是 ChatCut 的差异化点。它生成的是可调 props 的组件，用户能改 text、color、font、number、image、video。

OpenClacky 可以用：

Remotion component
SVG/HTML component
ASS subtitle style
JSON props

来复刻，而不是直接生成 mp4。

原则 9：Reference anchors 比纯 prompt 更稳定

ChatCut 文档明确说，上传品牌图、参考照片、截图能 anchor visual style 并显著改善结果。

OpenClacky 里要把 reference 做成一等公民：

{
  "reference_images": [],
  "brand_colors": [],
  "style_keywords": [],
  "layout_examples": []
}
原则 10：网页/品牌理解是 Agent 的重要能力

ChatCut 的 brand promo 案例非常关键：Agent 浏览网站、下载 logo/images、识别 key selling points 和 brand colors，然后选模板组装视频。

OpenClacky 可以直接复刻成：

web.extract_brand_assets
web.extract_product_claims
web.extract_visual_style
video.make_brand_promo

这很适合 OpenClacky，因为它本来就擅长浏览器/文件/命令/Skill 编排。

原则 11：按场景封装 Master Prompt

ChatCut 博客建议把重复视频格式写成 master prompt，比如客户案例、产品 demo、FAQ、社媒系列、播客访谈，因为这些内容结构稳定，适合自动化 first cut。

OpenClacky 可以把这些 master prompt 产品化为 Skill 模板：

video.customer_testimonial
video.product_demo
video.podcast_shorts
video.course_summary
video.ugc_ad
video.brand_promo
原则 12：专业工作流必须能导出

ChatCut 不只导出成片，还强调 transcript、SRT、DOCX、XML、ProRes 4444 alpha、NLE 工作流。

OpenClacky 应该支持：

final.mp4
captions.srt
captions.ass
transcript.docx
timeline.json
edit_decisions.json
premiere.xml / fcpxml / otio

这样才像专业剪辑助理，不是玩具。

一、ChatCut 做得好的部分
1. 不是“功能按钮”，而是 Agent-first

ChatCut 官方把自己定义成 web-based AI video editing agent：用户用自然语言告诉 Agent 想要什么，Agent 负责重排素材、找高光、生成字幕、生成动效、生成音乐。

这点是它最核心的产品差异：

传统视频编辑器：
用户自己找功能 → 自己点按钮 → 自己调时间线

ChatCut：
用户说目标 → Agent 理解素材 → Agent 调工具 → 时间线被修改

OpenClacky 应该直接抄这个产品抽象：
不要做“视频工具箱”，要做“视频剪辑 Agent”。

2. 真实多轨时间线，不是黑盒生成

ChatCut 一直强调它操作的是 real multi-track timeline。官网 demo 里能看到 MG、V1、A1、MUS 等轨道，Agent 会把 title card、quote lower-third、music bed 都落到真实时间线上。

这点非常重要。很多 AI 视频工具的问题是：

输入 prompt
→ 输出一个 mp4
→ 用户不满意只能重新生成

ChatCut 更像：

输入 prompt
→ 生成 timeline patch
→ 用户可以继续剪、改、撤销、导出

OpenClacky 要借鉴的是：

所有 Skill 都不要只输出文件
必须输出：
1. asset
2. timeline patch
3. edit report
4. rollback point
3. Transcript 就是 Timeline

ChatCut 最强的体验之一是 文本剪视频。官方写得很明确：每个词都有时间戳，删掉一句文字，对应的视频和音频也会被删掉。

字幕页也强调 word-level timestamps，可以做逐词高亮、精准裁切、文本剪辑、精准同步。

OpenClacky 最该先复刻这个：

video.mp4
→ ASR
→ word-level timestamp
→ transcript.json
→ 用户 / Agent 删除句子
→ 生成 cut list
→ ffmpeg 渲染

这比先做复杂 UI 更重要。

4. 适合口播、访谈、播客、课程

ChatCut 官方承认它现在的素材理解主要基于转写文本，最适合 talking head 和 interview footage；视觉分析还在 coming soon。

这其实反而给 OpenClacky 一个清晰入口：

不要一上来做全类型视频剪辑
先做：
口播视频
访谈
播客
课程
产品 demo
会议录制

这些场景的核心不是视觉理解，而是：

转写
删废话
找高光
重排段落
加字幕
导出短视频
5. Agent 会做多步骤任务链

ChatCut 不是只执行一个动作。官方示例里，Agent 会先 read_script，然后检查重复 take、filler、silence，再 apply_script 剪辑，接着加 motion graphics，再生成 music bed 并压在人声下面。

这说明它内部很像：

read_project
read_script
analyze
plan
apply_script
render_preview

OpenClacky 应该借鉴这个 Agent loop：

1. read_project
2. read_transcript
3. inspect_timeline
4. propose_edit_plan
5. apply_timeline_patch
6. render_preview
7. write_edit_report
6. Motion Graphics 做成可编辑组件

ChatCut 的动效不是简单生成一段视频。官方文档说它会生成 editable animation component，然后用户可以调颜色、文字、字体、时间、布局，最终还可以导出 ProRes 4444 alpha 透明通道。

这点非常值得抄。

OpenClacky 不要做：

prompt → 生成动效 mp4

应该做：

prompt → 选择动效模板 → 填 props → 渲染透明层 → 放进 timeline

例如：

{
  "type": "lower_third",
  "start": 3.2,
  "duration": 4,
  "props": {
    "name": "Yifan",
    "title": "OpenClacky",
    "style": "clean tech",
    "animation": "slide in left"
  }
}

底层可以用：

Remotion
SVG
HTML Canvas
ASS subtitle
FFmpeg overlay
7. URL → 品牌视频，这个方向很适合 Agent

ChatCut motion graphics 文档里有一个非常关键的能力：用户给 Zillow / Spotify / Apple / Lovable 这类官网 URL，Agent 会浏览网页、下载 logo 和图片、识别卖点和品牌色，然后选择 motion graphics template 组装 promo。

AI Video Generation 文档也说，Agent 能分析产品页，理解产品卖点，写 UGC ad 脚本，并生成 9:16 移动端广告。

这非常适合 OpenClacky，因为 OpenClacky 本来就有浏览器、文件、命令、Skill 的能力。

可以做成：

skill: video.brand_promo_from_url

输入：
官网 URL / 产品页 URL

执行：
1. 抓网页
2. 抽 logo / 色彩 / slogan / 产品截图
3. 总结卖点
4. 写 15 秒脚本
5. 生成 title card / lower third / CTA
6. 加字幕 / 配乐
7. 导出 9:16 / 16:9
8. Reference image 做成一等公民

ChatCut 文档明确说，上传品牌图、参考照片、截图可以 anchor visual style，显著改善结果。

OpenClacky 可以把 reference 也做成 Skill 输入标准：

{
  "references": {
    "brand_logo": "logo.png",
    "style_image": "reference.jpg",
    "color_palette": ["#000000", "#00ff66"],
    "example_video": "sample.mp4"
  }
}

这样视频 Skill 不只是听 prompt，而是有视觉锚点。

9. 计费透明，这点也值得借鉴

ChatCut 的 credits policy 写得很细：AI 动作消耗 credits，手动编辑、上传、自动转写、添加已有 transcript 的字幕、保存版本、timeline 导航不收费；Agent turn 的费用由模型、token、tool calls、cache reads/cache creations 影响。

这正好打中 OpenClacky 的核心叙事：省 token、可控成本、账单透明。

OpenClacky 可以比它更进一步：

每个 Skill 执行前预估：
- input tokens
- output tokens
- cache hit
- tool cost
- render cost
- expected runtime

执行后报告：
- 实际花费
- 缓存命中
- 每一步成本
- 哪一步最贵

这个会比 ChatCut 更贴 OpenClacky 的“省钱 Agent”定位。

二、ChatCut 做得差 / 可以被 OpenClacky 反打的部分
1. 不是开源，无法自定义底层工作流

ChatCut 是闭源 SaaS，用户只能用它给出的 Agent、模型、编辑器和 credits 系统。OpenClacky 可以反打：

ChatCut：好用，但黑盒
OpenClacky Video Skills：开源、可改、可接本地模型、可接自己的 ffmpeg / Whisper / Remotion / TTS

对极客和企业用户，这个差异很重要。

2. 视觉理解还没完全成熟

ChatCut 自己文档写了：当前 footage analysis 是 transcription-based content understanding，最适合 talking head / interview footage，visual analysis coming soon。

这意味着它目前更擅长“有说话内容的视频”，不是真正全视觉剪辑。

OpenClacky 可以避开硬仗，先明确定位：

第一阶段：
口播 / 访谈 / 播客 / 课程剪辑

第二阶段：
加 keyframe / scene detection / OCR / visual search

第三阶段：
真正视觉理解剪辑

不要一开始吹“任何视频都能剪”。

3. 上传限制和工程限制明显

ChatCut docs 写到 rough cut assembly 支持上传 raw footage，限制是 up to 10 clips, 5GB total。

它的 export 也有上限：官方 editor overview 写到最高导出 1080p 60fps；credits policy 也写 max resolution across all plans 是 1080p。

OpenClacky 可以反打：

本地 ffmpeg 渲染
不限制素材数量
不限制 1080p
能输出 4K / ProRes / DNxHR / XML / OTIO

当然这取决于本地机器性能，但至少产品叙事上有空间。

4. 浏览器 SaaS，网络依赖重

ChatCut 的卖点是 browser-based，但这也是弱点。它自己的博客也承认 browser-based 场景下慢网会拖慢工作。

OpenClacky 可以做 local-first：

素材在本地
转写可本地
剪辑可本地
渲染可本地
必要时才调用云模型

这对国内用户、企业素材、隐私视频非常有价值。

5. 成本会被 AI generation 和 agent turn 吃掉

ChatCut 明确说：AI 相关都消耗 credits，包括 Agent chat turns、视频生成、图片生成、motion graphics、音乐、旁白、render。视频生成按每秒 0.6 credits 计费，Agent turn 还看模型、token、工具调用、cache reads/cache creations。

OpenClacky 可以反打的点：

ChatCut：每次 AI 工作都进入 credits 体系
OpenClacky：Skill 本地跑，只有必要 LLM 调用收费

尤其是这些可以本地化：

静音检测
口癖删除
ffmpeg 裁切
字幕烧录
横竖屏导出
音量标准化
基础动效模板

这很符合你之前一直强调的：token 账单敏感用户。

6. 多人协作还没做好

ChatCut AI video editor FAQ 里写到，多 tab / view 会同步，但 real-time multi-user editing isn't supported yet，还在 roadmap。

OpenClacky 不一定要先做协作，但可以做：

project bundle
edit_decisions.json
timeline diff
version snapshot
Git-like versioning

这对开发者和企业团队更自然。

7. 模板库可能不如 Canva / CapCut 丰富

ChatCut 自己在 promo video maker 文章里承认，它比 legacy tools 更新，template library 比 Canva 小。

OpenClacky 可以不跟它拼模板数量，而是做 Skill 市场：

用户自己写 Skill
社区贡献模板
行业模板包
一键复用工作流

这会比封闭模板库更适合 OpenClacky。

8. AI 视频生成上限短

ChatCut AI Video Generation 文档写到，Seedance 2.0 生成 clip up to 15 seconds。

所以它适合做：

B-roll
转场
产品镜头
短广告素材

不适合直接生成长视频。

OpenClacky 复刻时不要把重点放在 text-to-video，而应该放在：

已有素材的自动剪辑
长视频切短视频
字幕 / 动效 / 配乐
URL → 广告视频工作流

四、你记得的开源项目：重点看这几个
1. FireRed-OpenStoryline：最值得 OpenClacky 借鉴

这个项目非常对口。它自称是 AI video editing agent，通过自然语言交互、LLM-powered planning、精准工具编排，把手动剪辑变成“意图驱动的导演式创作”；并且强调 human-in-the-loop 和 reusable Style Skills。

它的功能点包括：

1. 自动搜索并下载图片 / 视频素材
2. clip segmentation 和内容理解
3. 智能脚本生成
4. Few-shot 风格迁移
5. BGM / voiceover / font 推荐
6. 自然语言反复修改剪辑
7. 保存完整剪辑工作流为 custom Skill

GitHub README 里明确写了 Editing Skill Archiving：保存完整剪辑工作流为自定义 Skill，换素材就能复用风格，实现批量创作。

这点和 OpenClacky 简直天然匹配。

OpenClacky 可以直接借鉴它的思路：

不是只生成一个视频
而是把一次成功剪辑沉淀成 Skill

第一次：
“帮我做一个小红书 AI 工具介绍视频”

系统生成：
script style
字幕风格
BGM 风格
节奏规则
转场规则
封面规则

第二次：
换素材，直接套同一个 Skill

它甚至已经支持 Agent Skills，提供 openstoryline-install 和 openstoryline-use 两个 Skill，还写到可以通过 OpenClaw、Claude Code、Codex 等 Agent 使用。

给 OpenClacky 的借鉴句
OpenClacky 可以参考 FireRed-OpenStoryline，把视频创作从“一次性生成”变成“可归档、可复用、可迁移的 Editing Skill”。用户不是每次重新 prompt，而是把成功的视频工作流沉淀成可复用 Skill，换素材即可批量生产同风格视频。

OpenCut 是 open-source CapCut alternative，现在 GitHub 上有 61k+ stars。它的 README 里写到正在重写架构，计划包括：

这些点对 OpenClacky 很关键。

MCP server for AI agents

视频编辑器不是只有 GUI

这和 OpenClacky 很匹配。

给 OpenClacky 的借鉴句
OpenClacky 可以参考 OpenCut 的 plugin-first + headless + MCP server 思路，把视频编辑能力做成可被 Agent 调用的底层 runtime，而不是绑定在单一 UI 上。OpenClacky 负责自然语言规划和 Skill 编排，OpenCut 类 runtime 负责 timeline、preview、render、export。
3. VideoDB Director：适合借鉴多 Agent 编排

Director 是一个开源 video agents framework，定位是让 video agent 能处理 search、editing、compilation、generation，并且流式返回结果。README 里写到它有 backend reasoning engine、chat-based UI、video player、collection view。

它的 Reasoning Engine 设计很值得借鉴：

1. contextual understanding
2. dynamic agent orchestration
3. modular processing
4. multi-agent coordination
5. real-time updates
6. extensible design

README 里的例子是：

Create a clip of the funniest scene in this video and share it on Slack.

系统会编排 upload、scene detection、clipping、sharing agents。

给 OpenClacky 的借鉴句
OpenClacky 可以参考 Director，把视频任务拆成多个专用 Agent：TranscriptAgent、HighlightAgent、CaptionAgent、RenderAgent、ShareAgent。主 Agent 只负责任务拆解和编排，具体执行交给 Skill。
4. HKUDS VideoAgent：适合借鉴“意图分析 + 图工作流”

VideoAgent 是 2026 年 6 月出现的开源/论文项目，定位是 all-in-one framework for video understanding, editing, and remaking。它的系统概览强调三点：

Intent Analysis
Autonomous Tool Use & Planning
Multi-Modal Understanding

它会把用户指令拆成显性和隐性子意图，再通过 intent-to-agent mapping 激活对应能力；工具规划用 graph-powered workflow，节点是工具能力，边是执行依赖。

给 OpenClacky 的借鉴句
OpenClacky 的视频 Agent 不应该线性地猜下一步，而应该生成一个 editing workflow graph：每个节点是 Skill，每条边是依赖关系。比如先转写，再删停顿，再生成字幕，再做横竖屏导出。失败时只重跑失败节点，不重跑整个视频任务。

这个虽然 stars 少，但关键词很对：local-first、transcript-based、OTIO/EDL、Shorts workflow。

传播点	ChatCut 官方 LinkedIn 强调 motion graphics 是 code-based、fully editable、可透明导出；这比“生成一段死视频”更容易打动专业用户。	OpenClacky 做动效 Skill 时必须输出结构化组件，不要只输出 mp4。
ChatCut 做得好的部分
1. 它把“剪视频”变成 Agent 工作流

官网明确说：用户用自然语言描述剪辑，ChatCut 在真实多轨时间线上执行。页面里能看到 read_script、apply_script、剪 silence、加 motion graphics、生成 music bed、duck 到人声下面这些执行痕迹。

OpenClacky 要学的是这个：

用户指令
→ read_project
→ read_transcript
→ plan_edit
→ call_skill
→ apply_timeline_patch
→ render_preview
→ export

不是：

用户指令
→ 直接生成一个最终 mp4
2. 它抓住了“空时间线恐惧”

Reddit 那条正面评价本质上说的不是“ChatCut 剪得多专业”，而是：最痛苦的是从空时间线开始想结构。ChatCut 帮用户先生成一个能看的 rough cut，后面精修反而轻松。

这对 OpenClacky 很重要。OpenClacky 的视频 Skill 不该卖“最终成片”，而该卖：

把素材从 0 整理到 60 分。
把空时间线变成可修改的初剪。

这比“AI 自动剪大片”可信。

3. Transcript 直接绑定 Timeline

ChatCut 官方说“每个词都有时间戳，删一句话，视频和音频也一起离开时间线”。这是它最核心的交互。

OpenClacky 要借鉴的不是字幕功能，而是这个数据结构：

{
  "word": "然后",
  "start": 12.34,
  "end": 12.56,
  "speaker": "S1",
  "clip_id": "A001"
}

有这个，才能做：

删口癖
删停顿
删重复 take
按主题重排
按高光摘片段
4. 它不让生成内容脱离项目

官网写得很清楚：缺 clip、thumbnail、music bed，直接描述，ChatCut 生成后放到 timeline，不需要导出再导入。

OpenClacky 的 Skill 输出应该统一成：

{
  "asset_path": "music_bed.wav",
  "timeline_patch": "patch_003.json",
  "report": "added music bed under A1, volume -18 LUFS"
}

不要让 Skill 只吐一个文件。

5. Motion Graphics 方向很准

ChatCut 官方强调 motion graphics 是 code-based、fully editable、透明导出，不是生成死图死视频。

OpenClacky 要学这个。动效 Skill 应该输出：

{
  "component": "lower_third",
  "props": {
    "title": "OpenClacky",
    "subtitle": "Local-first AI Agent",
    "duration": 4,
    "position": "bottom_left",
    "animation": "slide_in"
  }
}

底层可以用 Remotion、SVG、HTML Canvas、ASS/libass、FFmpeg overlay。
不要一上来搞文生视频，那样成本高、不可控、不可编辑。

6. 它保留专业软件出口

ChatCut 官方 LinkedIn 说可以导出 XML 继续在别的软件里编辑；Antler 文章也提到它能导出到其他 NLE 或 DAW，保持专业工作流控制权。

OpenClacky 要借鉴：

final.mp4
transcript.json
captions.srt
captions.ass
timeline.json
edit_decisions.json
premiere.xml / fcpxml / otio

对专业用户来说，能接 Premiere / Resolve / Final Cut 比“我有个网页编辑器”更有说服力。

ChatCut 做得差的部分
1. 最终剪辑质量仍然会被专业用户质疑

Reddit 评论里有人明确说，剪点和音乐节奏不自然，有些镜头需要多停留。这个问题本质上是 AI 粗剪常见问题：能拼结构，但对节奏、情绪、停顿、音乐点的判断不稳定。

OpenClacky 要避免吹“自动成片”。更准确的表达：

自动生成可编辑 first cut。
不是替代 final cut。
2. 容易被社区认为是“AI 软广产品”

同一 Reddit 讨论里有人直接讽刺这是“shonky Claudeslop AI products”的推广套路。这个不是事实判断，但代表专业社区对 AI 剪辑工具的警惕。

OpenClacky 做宣传时别写：

AI 自动剪出专业大片

应该写：

把口播/访谈素材自动整理成可审计的 rough cut，保留剪辑决策和可回滚 timeline。

这更像工程产品，不像营销玩具。

3. 外部生成模型依赖会伤信任

Seedance 相关 Reddit 讨论里，有用户抱怨 ChatCut 没有 Seedance、不能生成，甚至称其为 scam；这个只能当作社区主观投诉，不可直接定性 ChatCut 欺诈。但它说明：如果卖点依赖第三方生成模型，额度、接口、模型可用性一变，用户会立刻不信任。

OpenClacky 应该把能力分层：

稳定本地能力：
转写、删静音、删口癖、加字幕、裁切、导出

外部模型能力：
生成 B-roll、图片、音乐、配音

失败策略：
降级、重试、换模型、输出未完成报告
4. SaaS credits 模式天然和“省钱”冲突

ChatCut pricing 页写到所有计划解锁 Pro 功能，区别主要是 monthly credit budget；AI 视频、图片、motion graphics、音乐、agentic editing 都在 credit 体系里。

OpenClacky 可以反打：

ChatCut：每次 AI 编辑都吃 credits
OpenClacky：确定性剪辑本地跑，只有必要步骤调用模型

尤其这些不该花 LLM 钱：

silence detection
filler word detection
ffmpeg cut
caption burn-in
aspect-ratio export
audio normalization
timeline diff
5. 浏览器云端对大素材不友好

ChatCut 的优点是 browser-based、无安装；缺点是素材上传、云端渲染、网络、隐私、额度都会变成限制。官网也写 long renders happen in the cloud。

OpenClacky 可以强调：

本地素材不出机器
本地 ffmpeg 渲染
项目状态全在文件里
适合企业内部视频、课程、访谈、客户素材
6. 协作能力还不完整

ChatCut FAQ 明确写：跨 tab 同步有，但 real-time multi-user editing 还不支持，在 roadmap。

OpenClacky 不一定要做多人实时协作，但可以做更适合开发者的：

timeline patch
edit decision log
version snapshot
git diff
rollback

这比实时协作更符合 OpenClacky 的工程气质。

你要找的开源项目

最像你说的那个项目，我判断优先是这几个。

项目	借鉴价值	适合 OpenClacky 学什么
FireRed-OpenStoryline	最像 ChatCut 的开源 Agent 方向。它自称 AI video editing agent，通过自然语言、LLM planning、tool orchestration，把手动剪辑变成意图驱动创作，并强调 reusable Style Skills。	学“把一次剪辑工作流沉淀成 Skill”。
OpenCut	开源 CapCut 替代品。公开 README 提到 Editor API、插件优先、Rust core、MCP server、headless mode、scripting tab。	学“视频编辑 runtime 怎么给 Agent 调用”。
Director / VideoDB	开源 video agents framework，能围绕 search、editing、compilation、generation 做 agent 编排。	学“多 Agent 编排视频任务”。
Vex	Reddit 上一个开源 terminal AI video editor，作者明确说重点不是“AI edits video”，而是 agentic harness：LLM 不拥有真相，project state 才是真相；它保存 timeline operations、artifacts、undo/redo。	学“项目状态是真相源，LLM 只是选工具”。
HKUDS VideoAgent	开源/论文方向，强调 intent analysis、autonomous tool use、multi-modal understanding。	学“把用户意图映射到工具图”。

如果只能给 OpenClacky 团队推荐两个：FireRed-OpenStoryline + OpenCut。

原因很简单：

FireRed-OpenStoryline：学 Agent / Skill 思路。
OpenCut：学 timeline / editor runtime / MCP / headless 思路。
最值得 OpenClacky 直接抄的设计
1. Project state 是真相源

借鉴 Vex 和 ChatCut：

LLM 不直接“记住”视频怎么剪。
project.json / timeline.json / transcript.json 才是事实。
LLM 只负责读状态、选工具、写 patch。

这是视频 Agent 的底层原则。

2. 所有操作都必须是 patch
{
  "op": "cut",
  "track": "V1",
  "start": 12.3,
  "end": 14.8,
  "reason": "filler + silence"
}

这样才能：

撤销
重做
复查
算成本
算节省时长
导出给专业软件
3. 成功剪辑要能保存成 Skill

这是 FireRed-OpenStoryline 最值得 OpenClacky 借鉴的点。它强调 reusable Style Skills，OpenClacky 应该把这个做成核心卖点。

比如用户做了一条“小红书 AI 工具介绍视频”，系统保存：

节奏规则
字幕样式
标题卡样式
BGM 风格
封面规则
导出规格
常用 prompt

下次换素材直接复用。

这比 ChatCut 更适合 OpenClacky，因为 OpenClacky 本来就是 Skill 生态。

4. 不做完整视频编辑器，做可被 Agent 操控的视频 runtime

OpenCut 的路线很关键：Editor API、plugin-first、MCP server、headless mode、scripting tab。

OpenClacky 应该告诉团队：

我们不是做一个新 CapCut。
我们做一个能被 Agent 调用的视频编辑 runtime。

核心接口：

read_project()
read_transcript()
apply_patch()
render_preview()
export_video()
export_xml()
save_as_skill()
给 OpenClacky 团队的原话
ChatCut 真正做得好的地方不是 AI 视频生成，而是把视频剪辑拆成 Agent 可执行的工程状态：transcript、timeline、asset、patch、preview、export。它解决的是空时间线和粗剪结构问题，不是最终剪辑审美问题。

社区评价也印证了这一点：用户认可它能快速给出 rough cut、删 repeated takes 和口癖，节省初剪时间；但专业用户仍然质疑节奏、剪点和 AI 产品宣传味。所以 OpenClacky 不要承诺“自动成片”，应该承诺“生成可审计、可回滚、可复用的 first cut”。

开源借鉴重点看 FireRed-OpenStoryline 和 OpenCut。前者学 Editing Skill / Style Skill，把成功剪辑工作流沉淀成 Skill；后者学 Editor API、plugin-first、MCP server、headless mode，把视频编辑能力变成 Agent 可调用 runtime。OpenClacky 的优势应该是本地优先、低成本、Skill 可复用、项目状态透明，而不是再做一个封闭 ChatCut。
